# Agent Runtime

How the GRC Agent actually runs a turn: the execution loop, loop-detection,
context handling, session handling, tool dispatch, and retrieval. This is a
map of the production runtime (`src/grc_agent/`) â€” not an experiment file.

## Invariants (the rules everything else obeys)

- **One native boundary.** `grc_native_adapter.py` is the only module that
  imports `gnuradio`. Everything else is pure Python over the Pydantic surface
  it exposes.
- **One parameter filter.** `runtime/param_filter.py::keep_param` is the single
  keep/drop authority for every model-visible parameter, in every tool.
- **No in-band control flow.** Model-visible strings carry no ALL-CAPS
  directives, behavioral commands, or procedural recipes. The system prompt is
  the only behavioral authority. Runtime interventions are either (a) facts
  injected as `user`-role `<runtime_directive>` messages, or (b) runtime
  decisions that stop a turn.
- **No information is silently cut in tool outputs.** The only transformations
  applied to model-facing payloads are removal of *redundant/dead* fields
  (e.g. `id` that mirrors `instance_name`, an always-empty `errors:[]`,
  `native_ok` that mirrors `validation.status`). Where size must be bounded
  (context compaction), truncation is always explicit and flagged with a
  sentinel â€” never a raw slice.

## 1. The execution loop â€” `toolagents_runtime.ToolAgentsRunner`

A turn = one user message â†’ a bounded `while True` loop in
`ToolAgentsRunner._run_turn_events` (`toolagents_runtime.py`).

Per iteration:
1. **Safety ceiling.** If `tool_rounds_used >= max_tool_rounds` (default 8,
   `MVP_TOOL_SURFACE.default_max_tool_rounds`), stop with a typed
   `SAFETY_CEILING` result. This is the hard bound on tool round-trips.
2. **Prune + compact history.** `_prune_completed_episodes` then
   `agent.compact_history()` run before the model call.
3. **Model step.** `chat_agent.step(messages, tool_registry, settings)` with up
   to 3 retry attempts. A retry injects a unique `system_salt` so the model
   provider does not return a cached (often degenerate/empty) response.
4. **Dispatch tool calls.** Each model tool call routes through the registry
   delegate â†’ `agent.execute_tool` â†’ the engine (inspect/query/change). Results
   are appended to `chat_history` as `tool_model` messages, then `continue`.
5. **Terminate.** When the model returns text with no tool calls, that text is
   the final answer and the turn ends with `{ok: true, ...}`. If every retry
   returns empty (no text and no tool calls), the turn ends with
   `{ok: false, error_type: "empty_model_response"}` and a factual
   "No response was generated by the model." text â€” never an empty terminal.

Backend-unreachable (`APIConnectionError`/httpx transport errors) is caught
once and surfaced as a typed `backend_unreachable` result â€” never an unhandled
exception, never `sys.exit()`.

## 2. Loop detection â€” repeated identical failing calls

Added because every safety-ceiling failure in the transcript review
(scenarios 06/15/17) was the same shape: the model re-sent a **byte-identical
failing `change_graph` batch** 2â€“6 times and never self-corrected.

Mechanism (`toolagents_runtime.py`, inside the dispatch loop):
- A per-turn dict `failing_change_signatures` keys a canonical JSON signature
  of each failing `change_graph` call's arguments.
- **2nd identical failing call** â†’ a *factual* `<runtime_directive>` user
  message is appended: "the last change_graph call used arguments identical to
  a previous failed attempt and produced the same error; repeating unchanged
  arguments will not change the result." This is a fact, not a command â€” it
  respects "no in-band control flow". The model sees it on the next iteration.
- **3rd identical failing call** â†’ the turn stops with `SAFETY_CEILING` to
  avoid burning the whole budget on a stuck loop.

Scope: only `change_graph` (the write tool) is tracked. Read tools are allowed
to repeat. This is the only "cap" the runtime applies to model behavior, and
it is explicitly allowed by project policy.

## 3. Context handling â€” `runtime/model_context.py` + `runtime/tool_context.py`

The model sees, each step, the output of `render_model_messages`:
1. **System message** â€” `build_system_prompt()` rebuilt every turn (so session
   id etc. stay current). Versioned (`__version__`); a stale
   `docs/MODEL_CONTEXT_BIBLE.md` fails the default test gate.
2. **Pruned history** â€” `_prune_completed_episodes`: the active episode (from
   the last *human* user message onward) keeps full tool activity; completed
   prior episodes keep only assistant text, and prior `<runtime_directive>`
   user messages are stripped (they were ephemeral).
3. **Optional reminder** â€” appended as a `user`-role `<runtime_directive>`
   message (never `system` after turn 1 â€” chat templates reject mid-stream
   system messages; never a custom role tag â€” small backends reject non-standard
   wire roles).

**Compaction** â€” `compact_chat_history` (`runtime/tool_context.py`): a one-pass
proportional sizer that, when total history exceeds `history_compact_budget`,
slices each `ToolCallResultContent` payload down to a proportional share
(floor 64 chars, per-payload cap `max_tool_result_chars` default 4000). Every
slice is marked with a `[TRUNCATED: was X, kept Y]` sentinel â€” truncation is
always explicit, never a raw cut. This is the *only* place tool output is
sized down, and it is a necessity of the finite context window (`num_ctx`
120000), not an information-removal policy.

**Wire-format role safety** â€” every runtime injection is `user`-role. There is
no `system`-role injection after the first turn and no non-standard role tags.

## 4. Session handling â€” `flowgraph_session.FlowgraphSession`

Owns one `.grc` flowgraph as a live `gnuradio.grc.core.FlowGraph`. The session
holds: `path`, the live `flowgraph`, `is_dirty`, a monotonic `state_revision`
counter (in-session identity), and the persisted file SHA-256 (cross-session
identity).

- **Atomic save** â€” temp file â†’ `fsync` â†’ `os.replace()` â†’ directory fsync,
  guarded by `fcntl.flock` on `.grc_agent/<name>.lock`; a backup is taken
  before each save. No partial writes are ever visible.
- **Integrity** â€” `file_integrity_state()` detects external modification
  (file changed on disk since load); `change_graph` refuses with
  `stale_revision` until the user reloads.
- **Transactions / rollback** â€” `transaction.capture_session_state` /
  `restore_session_state` use GRC-native `export_data`/`import_data` (not a
  file reload) so unsaved dirty edits survive a rollback. `change_graph`
  snapshots before the batch and restores on validation failure (unless
  `force=True`). `change_graph` output is minimal: `{"ok": true}` on success;
  `{"ok": false, "error_type", "errors":[...]}` on failure.

## 5. Tool surface & dispatch â€” `agent.GrcAgent`

Three model-facing tools (`runtime/model_context.MVP_MODEL_TOOL_NAMES`):
`inspect_graph` (read), `query_knowledge` (read), `change_graph` (write). No
internal/legacy tools are registered.

- `normalize_tool_call_arguments` â†’ `validate_turn_route` â†’ `validate_tool_call`
  â†’ `execute_tool`. Extra (hallucinated) args are silently dropped
  (`extra="ignore"` inbound); the wire shape is locked (`extra="forbid"`
  outbound).
- **Engine correctness gates in `change_graph`** (uniform, every call): the
  `type` enum auto-resolves from neighbor dtype on newly-added polymorphic
  blocks â€” running **after** `update_params` (and its `rewrite()`) so a port
  created by a same-batch `num_inputs` bump already exists when the neighbor
  dtype is read; an explicit `type` set (via `add_blocks` or `update_params`)
  is never clobbered. Duplicate block names are rejected; no-op edits are
  detected; validation errors carry causal hints (orphaned-port, IO-type
  mismatch â€” the IO hint resolves the *neighbor's* dtype from live ports, not
  by pattern-matching the error text); and `set_param` rejects invalid enum
  tokens / `${variable:NAME}` template literals that native `set_value` would
  otherwise silently revert.

## 6. Retrieval â€” `query_knowledge` (`runtime/inspect_graph.query_knowledge`)

Two domains, one sqlite-vec + `embeddinggemma:latest` pipeline:

- **catalog** (`runtime/search_blocks.search_blocks`): embeds the query â†’ KNN
  over the block index â†’ top-`k` hits rendered via
  `BlockDescription.to_payload` (Stage-A filtered: drops `hide='all'`,
  Advanced/Config category, `gui_hint`). Each hit: `block_id`, `params`
  (`dtype=default` / `enum=[...]=default`), `inputs`/`outputs`
  (`{id, domain, dtype}` â€” a stream port with no explicit id gets its positional
  key `"0","1",â€¦` synthesized), the `category` path, and a `distance` score.
  Default `k=5` (max 12).
- **docs** (`runtime/doc_answer.ask_grc_docs`): embeds the question â†’ KNN over
  **section chunks** (each markdown file split on `#` headings, body â‰¤256
  words) â†’ top-`k` chunks â†’ **one helper-LLM call**
  (`_generate_grounded_answer`, same Ollama chat model, grounded in the chunk
  text with inline source citations) â†’ the main agent receives
  `{answer, sources:[{path, distance}]}`. Default `k=3` (max 8).

  The helper-agent step is the concise-output stage: raw chunks go to the
  helper, the main model gets the helper's grounded summary plus source paths.

All `k` values and caps are configurable in `[agent.retrieval]`
(`config.RetrievalConfig`); a cross-field validator rejects `default_k > max_k`.

## 7. Where sizing actually happens (and where it does not)

| Layer | Sizes down? | How |
|---|---|---|
| Tool engine output | **No** (only redundant/dead fields removed: `id` that mirrors `instance_name`, always-empty `errors:[]`, `native_ok` that mirrors `validation.status`, empty-valued params) | `keep_param`, `render_parameter`, inspect-dump exclude |
| Catalog/docs retrieval breadth | Configurable `k` (maintainer decision) | `RetrievalConfig` |
| Context window | **Yes â€” flagged truncation only** | `compact_chat_history` proportional slice + sentinel |
| Model behavior | Repeated-identical-failing-call stop only | loop detection |

Nothing cuts information silently. The only hard truncation is context
compaction, and it is always flagged.

## 8. Design decisions (investigated â€” not bugs)

Recorded so the next maintainer doesn't re-litigate them:

- **`disabled` vs `bypass` validation asymmetry is native GRC**, not an agent
  bug. Native `flow_graph.is_valid()` reports a connected source's downstream
  port as "not connected" when the source is `disabled`, but valid when
  `bypass`. The system prompt therefore tells the model to use `bypass` (or
  `force`) for connected blocks; the validator is intentionally not "fixed".
- **`ask_grc_docs` is chunk-based, not file-based.** Retrieval returns â‰¤256-word
  section chunks fed directly to the helper-LLM call; this replaced an older
  file-reload design to eliminate silent truncation. The helper-agent
  (`_generate_grounded_answer`) step is retained.
- **The `set_param` enum check is a root-cause fix for the engine hole**
  (native `set_value` accepts any enum string and `rewrite()` silently reverts
  invalid tokens). It cannot prevent a model from *inventing* a bad token
  (e.g. `float_const/float`) â€” it makes that error loud so the model retries.
- **Scenario 06 non-convergence is model reasoning, not an engine defect.**
  `tests/test_agent_flow_engine_core.py::06_multiply_rewire` proves the engine
  reaches 06's goal state; auto-removing the orphaned source would be a silent
  transformation, so the engine intentionally leaves that to the model.
