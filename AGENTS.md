# AGENTS.md

Rules for AI coding agents working on this codebase. Direct, data-driven, zero fluff.

## Architectural Vision & Core Rules

- **Simplify First**: Lean towards simplifying, not complicating. If a feature or approach is ad-hoc, hardcoded, or not essential, **remove it**.
- **No Brittle Reinventions**: Reject complex manual implementations or from-scratch logic when reliable, standard libraries can replace them. Always opt for robust libraries to avoid reinventing the wheel — PydanticAI owns the agentic loop, tool dispatch, and message history; do not reimplement any of that here.
- **No Backward Compatibility**: Delete dead code completely. Do not write shims, dual-format persistence layers, or legacy bridges. Keep changes clean and direct.
- **No Assumed Reasoning Failures**: Do not assume task failures are solely due to LLM reasoning. Audit the execution harness for context flooding, poor prompt construction, hidden ad-hoc logic, or silent error message clipping. Correctness lives at the source.
- **Maximizing Context & No String-Based Clipping**: Do not enforce arbitrary context limits beyond what the backend actually supports. Never clip inputs or outputs (like error payloads or long tool results) using raw character slicing which breaks structured context.
- **Be Bold, Objective, and Grounded**: Base every decision on grounded, verified observations, never on assumptions. Ask for clarification when requirements are ambiguous or when a major decision (e.g. library selection, backend config) needs to be made.

## Architecture at a glance

Everything lives in the installable `grc_agent` package, under `src/grc_agent/`. It must never import from anywhere outside the package — the one accepted exception is `ingest.py` reading the `docs/wiki_gnuradio_org/` corpus, which lives in the repo at dev time but **is** shipped package data: `pyproject.toml`'s `force-include` bundles it into both the sdist and the wheel (as `grc_agent/docs/wiki_gnuradio_org/`), and `_paths.py`'s `docs_dir()` checks that bundled location before falling back to the repo path, so an installed package works without the source checkout. It is the docs *vector DB built from that corpus* that is genuinely not shipped — see "Vector DBs are built, not shipped" below.

| File | Role |
|--------|------|
| `adapter.py` | The **only** module that imports `gnuradio` — including from within `web.py`'s `grc_render` subprocess, which calls `adapter.get_gui_platform()`/`gui_application_cls()` rather than hand-duplicating `Platform`/`Application` construction inline (a prior version of `grc_render` did the latter, textually violating this rule despite running in a separate interpreter). Lazy singleton `get_platform()`. All GRC-facing logic: flow-graph load/save, parameter/port visibility filtering, block role classification, the `change_graph` mutation engine, catalog/docs vector RAG, `lite_web_search` (a lite.duckduckgo.com scrape used as the local fallback of pydantic-ai's `WebSearch` capability on providers without native search). |
| `agent.py` | Wires `adapter`'s functions into PydanticAI `Tool`s via `grc_tools()` (`inspect_graph`, `query_knowledge`, `change_graph`) plus provider-adaptive `WebSearch`/`WebFetch` capabilities (defined once here as `web_search_cap`/`web_fetch_cap`, imported by `web.py` and the tests), defines the system prompt, hosts the scenario harness used by integration tests. Its `MODEL`/`OLLAMA_V1` constants are deliberately fixed (not wired to `settings.py`) so the scenario harness stays reproducible run-over-run. |
| `web.py` | Starlette app: proxies/rebrands `agent.to_web()`'s chat widget, serves the GNU Radio dashboard (`panel.html`) and its `/grc/*` JSON API, builds the interactive chat `Agent`'s model from `settings.py`'s saved provider/model preference. Also owns the broadwayd/`canvas_app.py` subprocess lifecycle — spawn, signal-based teardown, and orphan reclaim (see "Process lifecycle" below) — and the `/grc/undo`/`/grc/redo` endpoints (see "Undo/redo" below), `/grc/health` (async provider connectivity check), and `/grc/apikey` (write API keys to `.env`). |
| `panel.html` | The dashboard page — plain HTML/CSS, no build step. |
| `panel.js` | The dashboard's logic (extracted from `panel.html`'s former inline `<script>`) — plain vanilla JS, no build step, served at `/grc/panel.js`. |
| `canvas_app.py` | The GTK canvas subprocess (spawned per `/grc/open`), embedded in the dashboard via Broadway (GTK's HTML5 backend). Binds a small HTTP control server (`/ready`, `/resize`, `/reload`) before the heavy GRC platform build so `web.py` can probe/drive it early. Isolates the flowgraph `DrawingArea` from the rest of GRC's own window chrome. Manual edits (drag, or any edit that doesn't fire a trackable GTK signal — see "Undo/redo" below) atomic-write to disk under the same `fcntl.flock` + lost-update hash guard convention as `adapter.change_graph`, push an undo snapshot, then ping `web.py`'s `/grc/reload`. Disables GRC's own native undo/redo actions at startup (`adapter.disable_native_undo_redo()`) so the two stacks can't fight over the same file. Imports `gi`/GTK directly (by design — the sole-importer rule above is about `gnuradio` specifically) but gets `Platform`/`Application`/`write_flow_graph_atomic` exclusively from `adapter.py`. `gi`, `gnuradio`, and `cairo`/pycairo (also imported by `web.py`'s `grc_render` subprocess) are all system-provided via `--system-site-packages` (see README) and deliberately absent from `pyproject.toml`'s dependencies — not pip-installable independent of the system GTK/GNU Radio stack. |
| `ingest.py` | Builds the catalog/docs sqlite-vec databases from scratch on first use (`adapter._ensure_db_built`) — no separate CLI or warmup step. |
| `settings.py` | Persisted chat-agent preferences — active provider, per-provider chat model names, and cloud API keys — all stored in `.env` (the single source of truth, editable from the GUI's settings panel or `/grc/settings`/`/grc/apikey`, or by hand). `env_path()` resolves which `.env`: `GRC_AGENT_ENV` override → a `.env` found walking up from CWD (dev) → `~/.config/grc_agent/.env` (installed package). `load_settings()` reads the `.env` *file* (saved state); `web.py`'s module-level `_cfg = load_settings()` captured once at import is the running state — that split is what the dashboard's restart-required badge compares. |
| `_paths.py` | Package-relative runtime-data directory resolution (`vectors_dir()`, `docs_dir()`), each overridable via env var. |

Data flow: `.grc` file → `adapter.load_flow_graph()` → live `gnuradio.grc.core.FlowGraph` → `inspect_graph()` → JSON tool result.

## Engineering Rules

- **No hand-picked heuristics.** No per-field allowlists, per-scenario branches, regex routing, or prompt folklore. If logic is needed, it is one uniform rule applied to every case (see `keep_param` in `adapter.py`).
- **Prefer native methods.** Use GNU Radio GRC's Python API — `param.hide`, `param.category`, `Block.is_variable`, `flow_graph.is_valid()`, etc.
- **Fix at the source.** Correctness lives in the tool/handler that produces data, not in a post-processor.
- **No silent transformation.** Any truncation, filtering, or omission in model-facing output must be explicit.
- **Simplify by removal.** Prefer deleting code over adding it.
- **Evidence before assertions.** Every claim cites a verified observation, never intent. A green test is necessary, not sufficient — inspect actual data flow. When verifying UI behavior, drive it with a real (even headless) browser — curl/API-level checks can miss pure client-side rendering bugs.
- **Prefer pydantic_ai's own sanctioned extension points** (capabilities, `ModelRetry`, `AsyncTenacityTransport`, etc.) over hand-rolled retry/loop/context logic. Check what the framework already provides (e.g. via its current docs) before building something custom.

## Tool Surface

Five model-facing tools. Three are wired in `agent.py`'s `grc_tools()`, registered as plain
`Tool(fn, name=..., description=...)` relying on pydantic_ai's own signature/type-hint
introspection (Pydantic models `BlockAdd`/`ParamUpdate`/`StateUpdate` for `change_graph`'s
structured args) — there are no hand-written JSON schema constants. The other two are
provider-adaptive `WebSearch`/`WebFetch` capabilities (`web_search_cap`/`web_fetch_cap` in
`agent.py`), added to `capabilities=[...]` alongside `ProcessHistory`/`StopGracefully` rather
than being part of `grc_tools()`. Both are eager (`defer_loading=False`), so they are always
callable — no `load_capability` round-trip. On providers with native support (OpenRouter via
its plugins) search/fetch run server-side; on providers without it (Ollama has none) they
fall back to `local` — `adapter.lite_web_search` (a lite.duckduckgo.com scrape) for search,
and the bundled markdownify tool (`WebFetch(local=True)`) for fetch.

| Tool | Direction | Engine |
|------|-----------|--------|
| `inspect_graph` | read | `adapter.inspect_graph()` (Stage A + B filtered) |
| `query_knowledge` | read | `adapter.query_catalog()` (catalog domain) **or** `query_docs()` (docs domain, vector RAG) — builds its `.db` on first use via `ingest.py` if missing |
| `web_search` | read | pydantic-ai `WebSearch` capability — native on OpenRouter, `adapter.lite_web_search` (lite.duckduckgo.com scrape) local fallback on Ollama. Returns raw snippets for in-context grounding (no separate LLM synthesis call). |
| `web_fetch` | read | pydantic-ai `WebFetch` capability — native where supported, bundled markdownify fallback (`WebFetch(local=True)`) otherwise. |
| `change_graph` | write | `adapter.change_graph()` — 7-phase transactional batch mutation with rollback; a failure raises `ModelRetry` (see Key Conventions) instead of just returning `ok=false` for the model to notice on its own |

No internal tools, no legacy tool registry. Tool schema and system-prompt tuning are permitted for general fixes; no ad-hoc or hardcoded prompt/schema rules targeting specific test scenarios or block instances.

## Key Conventions

- **Param filtering** (one rule, in `adapter.py`'s `keep_param`): Stage A (every mode) drops `dtype == "id"`, `showports`, `bus_structure_*`, `hide == "all"`, `dtype == "gui_hint"`. Stage B (overview mode only) keeps `hide == "none"` OR `dtype == "enum"` with a non-default value or structural status OR `value != default` OR references a variable OR the param is type-controlling (native-derived via `type_controlling_params`, never a hardcoded name) OR `generate_options`. Do not reimplement filtering inline. A `category ∈ {Advanced, Config}` hard drop was tried and removed: GNU Radio files params as human-authored as `comment` and as clearly customized as a qtgui curve's color/marker/legend under those same two categories, so the blanket drop silently hid genuine non-default customizations (live-confirmed: a `note` block's entire explanatory text, and `qtgui_time_sink_x`'s customized `marker`/`color` values, were always invisible regardless of Stage B's own non-default rule). No fixture in the test corpus has an Advanced/Config param with `hide == "none"`, so removing the category check does not cause unconditional flooding — Stage B's existing per-param rules are sufficient on their own.
- **State values:** `enabled`, `disabled`, `bypass` (accept `bypassed` as alias). Validated against the block's own `STATE_LABELS`, not a hardcoded set.
- **Block lookup:** use native `flow_graph.get_block(name)`, not a manual scan.
- **Atomic save:** temp file → fsync → `os.replace()` → directory fsync. Lock via `fcntl.flock` on `.grc_agent/<name>.lock` next to the target file. Backup saved before each save.
- **The active flowgraph's state lock is shared with the tools, not just open/close/reload.** `FlowgraphProxy`'s `_flowgraph_state_lock` (`web.py`) is exposed via `get_state_lock()` and acquired by `agent.py`'s `inspect_graph`/`change_graph` tool functions (via the `hasattr`-guarded `_with_state_lock` helper — a no-op passthrough when `ctx.deps` is a raw flowgraph, e.g. the scenario harness, which has no lock). Without this, `FlowgraphProxy.__getattr__`/`__setattr__` doing a fresh `_target` lookup per attribute access meant a concurrent `/grc/open`/`/grc/close` swap could make a single tool call silently straddle two different flowgraph objects mid-mutation. `grc_open` releases this lock **before** the (up to 20s) canvas-readiness wait — that poll only talks to a separate process over a fixed control port and touches no shared state, so holding the lock across it would stall every other lock-serialized operation for the full timeout on a single slow/crashed canvas (live-reproduced: a concurrent `/grc/close` blocked for ~20s until this was fixed to return in under 100ms).
- **Process lifecycle (broadwayd/`canvas_app.py`):** Broadway's display number is derived from `BROADWAY_PORT` (`f":{BROADWAY_PORT}"`), never hardcoded — two instances on different ports previously collided on the same hardcoded `:5` Broadway unix socket, silently cross-connecting one instance's canvas to another's broadwayd. Both broadwayd and `canvas_app.py` get their own port-scoped pidfile (`_broadway_pidfile()`/`_canvas_pidfile()`), reclaimed on the next launch via a `/proc`-verified identity check (`_proc_comm` for broadwayd's `comm`, `_proc_cmdline` for canvas since it runs as a generic `python3` process) before being killed via `os.killpg` (both are `setsid`-detached process-group leaders) with a SIGTERM→wait→SIGKILL fallback (`_killpg_with_fallback`). `_spawn_broadway()` verifies the new process actually stayed running before committing its pidfile, so a failed spawn can't silently supersede and orphan a still-live prior instance. `atexit.register(_cleanup_procs)` alone does **not** cover SIGTERM: uvicorn's `capture_signals()` restores the pre-existing signal disposition and re-raises the signal after its own graceful HTTP shutdown — SIGINT's re-raise hits Python's built-in default handler (→ `KeyboardInterrupt` → normal exit → atexit runs), but SIGTERM has no such built-in handler, so its re-raise would hit the raw OS default (immediate kernel-level termination, atexit skipped) without the explicit `signal.signal(signal.SIGTERM, _handle_sigterm)` registered at module level in `web.py`. `/grc/open`'s response surfaces `canvas_ready`/`canvas_error` instead of unconditionally reporting success — the flowgraph itself loading (`ok: true`) is independent of whether the canvas subprocess actually became ready in time.
- **`openGraph()` (panel.js) must never route through `startNewConversation()`'s unload guard.** `startNewConversation()` closes any already-loaded flowgraph first if `state.isGrcLoaded` is true (needed for the "+" button/Clear History, which are genuinely abandoning the current file) — but `openGraph()` calling it after a successful `/grc/open` was a live-reproduced, deterministic bug: the always-running 750ms `pollConversationState` reliably notices the server-side version bump (which happens early in `grc_open`, well before the canvas spawn + readiness wait that makes the request itself take over 750ms) and flips `isGrcLoaded` true *while `openGraph`'s own fetch is still in flight* — so by the time `openGraph` reached the unload-guarded call, it closed the file it had just opened, every time. `openGraph` now calls the extracted `resetChatFrame()` (just the iframe-reset lines, no unload guard) instead. Relatedly, `openGraph`/the auto-open path in `pollConversationState` both check `data.canvas_ready`/`data.canvas_error` (see the `/grc/open` note above) and skip revealing the canvas iframe when false, rather than pointing it at a display with no live GTK client.
- **`change_graph` output:** `adapter.change_graph()` returns `{"ok": true}` on success; `{"ok": false, "error_type": "...", "errors": [...]}` on failure. `force=True` bypasses native-validation failures only — adapter errors (unknown param, missing block) cannot be bypassed. A batch with every operation array empty/absent is rejected rather than trivially returning `ok=true` with nothing applied. The `change_graph` **tool** wraps this: on `ok=false` it raises `ModelRetry` (with the errors folded into the message) instead of returning the payload as an inert tool result, so the model gets an automatic corrective turn — `change_tool.max_retries = 3`, pydantic_ai's own sanctioned retry budget. A separate `args_validator` (`validate_change_graph_args`) pre-checks that referenced block names actually exist before the call ever reaches `adapter.change_graph()`, raising `ModelRetry` early and cheaply for that specific failure mode.
- **Graph-validity gate:** `agent.output_validator(validate_flowgraph_state)` runs on every turn's final output. If any `change_graph` tool call appears in that turn's message history, it checks the live flow graph's native `is_valid()`/`iter_error_messages()` and raises `ModelRetry` with the real GNU-Radio-native error text if invalid — the model cannot end a turn having left the graph broken without either fixing it or explicitly using `force=True` earlier in the same turn.
- **Ports** (`inspect_graph`'s block `inputs`/`outputs`): Stage A drops hidden ports (native `active_sinks`/`active_sources`, already filtered). Stage B additionally drops a port only if it is both `optional` and unconnected; required or connected ports always show.
- **`type`/dtype auto-resolution:** `change_graph` accepts the literal string `"auto"` on a type-controlling param, resolved from a connected neighbor's dtype via `resolve_auto`. Known gap: resolution for a brand-new block whose only connection is in the same batch's `add_connections` list can silently fall through to GNU Radio's own default instead of resolving correctly — verify with a live-connected-block test case before trusting this path for new blocks.
- **The active flowgraph is a swappable proxy** (`FlowgraphProxy` in `web.py`), not fixed at startup — every tool already does plain attribute access on `ctx.deps`, so this needed no changes to `agent.py`'s tool code to support loading a different file mid-session. It starts empty (`None`) — a tool call before a file is loaded raises a clear `RuntimeError`, not a crash.
- **Vector DBs are built, not shipped — deliberately, since the catalog DB is machine-specific.** `src/grc_agent/vectors/*.db` is gitignored; `adapter._ensure_db_built` lazily calls `ingest.py` to build it on first `query_catalog`/`query_docs` call. Each backend (Ollama/OpenRouter/Ollama Cloud) gets its own `.db` file. Shipping a pre-built catalog DB was considered and rejected: it's derived from the user's own installed GNU Radio block library (~570 blocks, including any OOT modules), which varies by machine and GNU Radio version — a DB built on one dev machine would misrepresent another user's actual catalog. `ingest.py` stores two identity keys in a `_db_meta` table at build time, and `_ensure_db_built` checks both (plus vector dimension) on every query, auto-deleting + rebuilding on any mismatch: `embedding_model` (different models produce different embedding spaces) and `corpus_version` — `gr.version()` for the catalog domain, a content hash of `docs/wiki_gnuradio_org/` for the docs domain (both computed by `adapter._corpus_version`, cached per-process). Without `corpus_version`, a DB that still matched on `embedding_model` alone would silently keep serving stale results forever after a GNU Radio upgrade or a docs-corpus update shipped in a new grc-agent release — there was no error, just quietly wrong data. DBs from before this check was added (no `corpus_version` key) are treated as a mismatch and rebuilt once, same as any other staleness. Embedding routing: `ollama` and `ollama_cloud` both use local Ollama (`localhost:11434/v1`, `embeddinggemma:latest`) for embeddings — Ollama Cloud's API doesn't expose `/v1/embeddings`. `openrouter` uses `openrouter.ai/api/v1` with `OPENROUTER_EMBEDDING_MODEL` (default `perplexity/pplx-embed-v1-0.6b`). `adapter.embed_query`/`embed_document` share a `_embed()` helper that turns a bare connection failure into an actionable message (whether to check `ollama serve`/`ollama pull` or `OPENROUTER_API_KEY`, based on which endpoint was being reached) instead of surfacing the SDK's generic "Connection error."
- **Chat model/provider is user-configurable, not hardcoded.** `web.py`'s interactive agent builds its model from `grc_agent.settings`'s `load_settings()`, which reads the **`.env` file** (the single source of truth for all GUI-set preferences — provider, per-provider model names, and cloud API keys — editable via the GUI's settings panel, `/grc/settings`, `/grc/apikey`, or by hand). Three providers: `ollama` (local, `OllamaModel`+`OllamaProvider` at `localhost:11434`), `ollama_cloud` (cloud, `OllamaModel`+`OllamaProvider` at `https://ollama.com/v1` with `OLLAMA_CLOUD_API_KEY`), `openrouter` (cloud, `OpenRouterModel`). The active provider is `GRC_PROVIDER` (default `ollama`); the per-provider chat models are `OLLAMA_CHAT_MODEL` (default `qwen3.6:35b-a3b-q4_K_M`), `OLLAMA_CLOUD_MODEL` (default `deepseek-v4-flash:cloud`), `OPENROUTER_MODEL` (default `deepseek/deepseek-v4-flash`). A saved change requires an app restart — no live in-process model swap; `save_settings()` writes only to the `.env` file and deliberately does NOT touch `os.environ`, so the dashboard's restart-required badge keeps correctly distinguishing saved from running. The dashboard's model-name field is click-to-edit. `agent.py`'s scenario-harness `MODEL`/`OLLAMA_V1` constants are intentionally separate and fixed, for reproducible benchmarking.
- **Provider health and API key management.** `GET /grc/health` async-checks connectivity for the current provider: pings `localhost:11434/api/tags` for Ollama, `https://ollama.com/v1/models` for Ollama Cloud, `https://openrouter.ai/api/v1/models` for OpenRouter. The dashboard calls this on page load and on every provider switch, showing a green/red badge — never blocks the UI. `POST /grc/apikey` writes an API key to `.env` for `ollama_cloud` or `openrouter` (also sets it in the current process so the health check sees it immediately). A key button appears in the model selector overlay when a cloud provider's key is missing.
- **`flow_graph.validate()` must be called before trusting `is_valid()`/`iter_error_messages()`.** GNU Radio's `Element._error_messages` list is populated only by an explicit `.validate()` call and is cleared — without being refilled — by `.rewrite()`. Since every mutation path calls `rewrite()`, skipping the `validate()` call right before checking `is_valid()` makes the check vacuously pass regardless of the graph's real state (live-confirmed: removing a required connection without `force=True` was silently accepted and persisted to disk). Both `inspect_graph()` and `change_graph()` call `flow_graph.validate()` immediately before their own `is_valid()` gate — do not remove either call as apparently-redundant.
- **Undo/redo is a disk-based snapshot stack, shared across both mutation sources.** `adapter.py`'s `push_undo_snapshot()`/`undo_flowgraph()`/`redo_flowgraph()`/`undo_status()` maintain `.grc_agent/<name>.undo/` (numbered `NNNNN.grc` snapshots + a `cursor.json` of `{index, count, hash}`), guarded by the same `fcntl.flock` as the main save. Both `adapter.change_graph()` (the agent's edits) and `canvas_app.py`'s `_do_trigger_reload()` (manual canvas edits) call `push_undo_snapshot()` right after their own atomic write, so one shared history covers edits from either process. A push dedups against a genuinely-unchanged state via content hash (`flow_graph_content_hash`, in-memory-serialization-based, not raw file bytes — see `canvas_app.py`'s `last_synced_export_hash` note below for why that distinction matters), and truncates any redo branch beyond the current cursor, standard undo/redo semantics. `undo_flowgraph`/`redo_flowgraph` never touch an in-memory flow graph directly — callers (`web.py`'s `/grc/undo`/`/grc/redo`) reload from disk afterward, same as `grc_reload` already does for a canvas-triggered disk change. `undo_status()` is a cheap, lock-free read used by `/grc/status` on every poll so the dashboard's Undo/Redo buttons stay in sync without an explicit refresh action. GRC's own native undo/redo actions are disabled in `canvas_app.py` (`adapter.disable_native_undo_redo()`) so a user can't drive GRC's separate, unsynced undo stack out from under this one.
- **New blocks are positioned by a headless, collision-avoiding placement — never by GRC's own native "add block" logic.** GRC's GUI action for this (`gui/canvas/flowgraph.py`'s `add_new_block`) just drops the block at a random point inside the current GTK scroll viewport with no collision check at all, and its one real anti-overlap logic (`paste_from_clipboard`'s grid-aligned nudge loop) depends on `gui.Constants.CANVAS_GRID_SIZE`, which imports `gi.repository` — a GTK dependency the headless agent-edit path must not take on. A block's real pixel size is also GUI-only (computed from Pango text metrics at draw time in `gui/canvas/block.py`); `core.Block` carries no width/height at all, so no code path, native or otherwise, can know a block's true footprint headlessly. `change_graph`'s `add_blocks` phase uses a connection-aware spiral grid search (`_find_block_placement`): each new block's target point is computed from its connected neighbors in the same batch's `add_connections` (placed to the right of the average neighbor position, matching L-to-R signal flow), or the graph's centroid if it has no connections. A spiral search then expands outward on a grid (`BLOCK_FOOTPRINT_W`/`BLOCK_FOOTPRINT_H`/`BLOCK_SPACING`) to find the nearest non-overlapping slot via proper AABB collision (`_rects_overlap`), filling empty space instead of always extending to the right. This must stay fully automatic, since `inspect_graph`'s model-facing output filters coordinates out of context entirely, so the agent has no way to reason about or request a position itself.
- **An unknown param name's error lists the block's actual valid param keys** (`set_param`, mirroring the sibling enum-value error's existing "not one of the valid options [...]" pattern) — live-confirmed a model spent an extra `query_knowledge` round-trip after guessing `samp_rate` instead of `qtgui_time_sink_x`'s actual `srate`, because the old error only said the guessed name was wrong, not what the right one was.
- **`resolve_auto` only ever resolves from an explicit, non-`"auto"` value — it never guesses from an equally-unresolved neighbor, and it fails loudly instead of silently defaulting.** Previously, two brand-new blocks connected to each other in the same batch, both left `"auto"`, fell through to reading each other's *own untouched schema default* and treated that as "resolved" — confirmed live: `analog_sig_source_x` + `qtgui_time_sink_x` both default to `complex`, so this looked like a working resolution purely by coincidence; a pair with genuinely different defaults (verified: `analog_sig_source_x` vs `dtv_dvbt_ofdm_sym_acquisition`, `complex` vs `float`) silently produced a real dtype mismatch, caught only by luck via the final `validate()` gate. `resolve_auto`'s terminal fallback (and its block-not-found path) now raises instead, surfaced through Phase 5's existing `auto_resolve_failed` error — the model gets an actionable "set an explicit type on at least one side" message instead of a silent guess. Explicit-value propagation (from either an existing live neighbor or an explicit value elsewhere in the same batch) is unaffected and still works exactly as before. A related, independently-confirmed bug in the same area: Phase 5 (`change_graph`'s `auto_resolve_types` loop) used to treat *any* param literally valued `"auto"` as needing type-resolution, not just actual type-controlling ones — some blocks have unrelated params whose own schema default happens to be the string `"auto"` too (e.g. `blocks_throttle2`'s numeric `limit`), which have no ports to resolve from at all. The loop now checks `type_controlling_params(block.key)` membership first.
- **`set_param`'s "unknown param" error lists the block's actual valid param keys** (mirrors the sibling enum-value error's existing "not one of the valid options [...]" pattern) — live-confirmed a model spent an extra `query_knowledge` round-trip after guessing `samp_rate` instead of `qtgui_time_sink_x`'s actual `srate`, because the old error only said the guessed name was wrong, not what the right one was.
- **`change_graph`'s tool description is derived from its own docstring (`docstring_format="google", require_parameter_descriptions=True`), not a separately hand-maintained string.** An explicit `Tool(description=...)` replaces a function's docstring rather than supplementing it — this tool used to carry both, and the hand-written one had already drifted from real behavior (no mention of `force`, no phase-order or auto-resolution notes). `inspect_graph`/`query_knowledge` still pass an explicit `description=` that merely duplicates their own (undrifted) docstrings verbatim — flagged as a smaller, not-yet-addressed version of the same redundancy.
- **A properties-dialog or context-menu edit in the canvas never fires GTK's `window-added` signal.** GRC's `PropsDialog` is a plain `Gtk.Dialog(transient_for=parent)`, never registered via `Gtk.Application.add_window()` — so the hook that normally detects "a new window opened, watch for its close" silently never sees it, for OK/Apply/Cancel alike, and the same is true of context-menu actions (Delete/Cut/Rotate/Enable/Disable/Bypass). `canvas_app.py` covers this with a periodic safety net (`_check_for_unsynced_edit`, `GLib.timeout_add(1500, ...)`) that re-serializes the in-memory graph and compares it against `last_synced_export_hash` — deliberately a separate field from `last_disk_hash` (raw file bytes), since GRC's own reload round-trip fills in defaults (e.g. `rotation: 0`) that make a raw-bytes comparison false-positive on every reload. Do not add more individual GTK signal hooks for this — it's the same "no hand-picked heuristics" violation the polling safety net was written to avoid.
- **`isolate_drawing_area` skips `Gtk.ScrolledWindow`/`Gtk.Notebook` ancestors entirely rather than hiding their siblings — defensive, not the actual fix for panning.** It hides every sibling on the path from the `DrawingArea` up to the window (menu bar, toolbar, notebook tabs, ...) so only the flowgraph canvas shows. Direct introspection (`ScrolledWindow.get_children()`) confirmed this GTK3 binding's `ScrolledWindow` only ever returns `[Viewport]` from `get_children()` — its scrollbars are *not* among them, so the blanket sibling-hiding was never actually hiding them in the first place; the skip added here is defensive (in case a different GTK3 minor version's binding does include them) rather than the mechanism that fixed panning.
- **The real reason the canvas couldn't be panned/resized to fit the dashboard's actual pane: GRC's own `Notebook.py` hardcodes both the `ScrolledWindow` and the `Notebook` wrapping the canvas to a 600×400 minimum size request.** A GTK container can never be allocated smaller than its child's requested minimum, so `apply_resize()`'s `window.resize(width, height)` calls were silently clamped to at least 600×400 no matter what the dashboard's real (often narrower) pane asked for — live-confirmed via direct introspection (`window.get_size()` after a resize request below 600 still reported 600+, and the adjustment/scrollbar state was otherwise entirely correct: `visible=True`, `mapped=True`, `upper > page_size` whenever content genuinely exceeded the window). The canvas then always rendered wider/taller than the visible iframe, silently cropping content off past the right/bottom edge with no scrollbar reachable within the visible area to recover it. Fixed by calling `set_size_request(1, 1)` on both the `ScrolledWindow` and the `Notebook` ancestor inside `isolate_drawing_area` — confirmed via introspection that `window.resize()` requests below 600×400 now actually apply (down to values as small as 400×476 tested live). GRC's own native `gui/canvas/block.py`'s "GTK-CRITICAL: gtk_box_gadget_distribute: assertion 'size >= 0' failed" warning (present in `canvas.log` on every reload before this fix) also stopped appearing once the `Notebook`'s own minimum was relaxed alongside the `ScrolledWindow`'s.
- **The chat UI is a native widget, not the pydantic-ai CDN iframe.** `web.py` keeps `agent.to_web()`'s stateless `/api/chat` backend (Vercel AI SDK UI Message Stream `v1` protocol — `data:{json}\n\n` SSE frames; the client owns full `messages[]` history and sends it each request) but serves its own dashboard at `/grc/panel` instead of proxying the `@pydantic/ai-chat-ui` CDN bundle. `panel.js`'s `initChatWidget`/`sendChatMessage`/`_consumeSSEStream` POST to `/api/chat` and render `text-delta`/`tool-input-start`/`tool-output-available`/`done`/`error` frames directly — no iframe, no CDN, no overlay-positioning, works fully offline. `/` redirects to `/grc/panel`. The old `chat_ui_index`/`chat_ui_asset`/`CHAT_UI_CDN_BASE`/`_asset_cache` shim and the `integrateSettings`/`integrateModelSelector`/`integrateNewConversationButton` overlay hacks are deleted; the model/provider/health/API-key/Clear-History controls live in a native `#chat-toolbar` at the top of `#chat-pane`.
- **Middle-click (button 2) canvas panning has no conflict with GRC's own canvas handlers — verified by source inspection, not assertion.** `canvas_app.py`'s `on_button_press`/`on_motion_notify`/`on_button_release` bind button-2 drag-to-pan on the `DrawingArea` and return `True` to consume. GRC's own `gnuradio/grc/gui/DrawingArea.py` (`_handle_mouse_button_press`/`_release`/`_motion`) only branches on button 1 (select/drag-move) and button 3 (context menu); there is no button-2 path and `button_state[2]` is set but never read. GTK's run-last emission runs the native handler first (returns not-handled for button 2), so emission continues to the custom handler, which owns the event cleanly. No double-fire, no native autopan.
- **`_do_trigger_reload` skips no-op clicks via the same content-hash gate the 1.5s safety-net poll uses.** `on_button_release` calls `trigger_reload()` for *every* non-middle-click release (including a plain select-click that changed nothing). The gate at the top of `_do_trigger_reload` returns early when `flow_graph_content_hash(drawing_area._flow_graph) == ctx.last_synced_export_hash` — without it, every left-click wrote the file, pushed an undo snapshot, bumped the version, and forced a web-side reload + canvas refresh.
- **A newly-added block can land far outside the visible viewport, with nothing to indicate that — `apply_reload` scrolls it into view.** `change_graph`'s `add_blocks` positioning (see the entry above) places new blocks in a column that grows further right with each edit; without an explicit cue, a user has no way to know a block was added unless they happen to zoom out or scroll there themselves (Ctrl+scroll for zoom still worked, since it's DrawingArea-native — panning was blocked by the 600×400 floor above). `apply_reload()` now diffs the block set before/after reloading from disk, and for any newly-appeared block, sets the `ScrolledWindow`'s adjustment values to bring it into view. This requires computing the flowgraph's own content extent directly (`flow_graph.get_extents()[2:]`, the same formula `DrawingArea._update_size` uses) and calling `adjustment.set_upper()` on it explicitly, rather than calling `_update_size()` and reading the adjustment's `upper` afterward — live-confirmed that `_update_size()` correctly computes a wider `size_request`, but propagating that into the adjustment's actual bound requires a real GTK size-allocate cycle that does **not** happen synchronously, nor within a couple of `GLib.idle_add` hops afterward; the adjustment stayed at its stale, pre-edit bound in both cases, silently truncating the scroll target back toward wherever the old bound was. Setting the bound directly sidesteps that timing uncertainty entirely.
- **A newly-added block can land far outside the visible viewport, with nothing to indicate that — `apply_reload` scrolls it into view.** `change_graph`'s `add_blocks` positioning (see the entry above) places new blocks in a column that grows further right with each edit; without an explicit cue, a user has no way to know a block was added unless they happen to zoom out or scroll there themselves (Ctrl+scroll for zoom still worked even with the scrollbar bug above, since it's DrawingArea-native — panning didn't). `apply_reload()` now diffs the block set before/after reloading from disk, and for any newly-appeared block, sets the `ScrolledWindow`'s adjustment values to bring it into view. Setting the bound directly sidesteps that timing uncertainty entirely.
- **A canvas timeout must never look like "nothing loaded" — several gaps let the two get conflated in practice, all fixed together.** `grc_open`'s contract (`web.py`, see "Process lifecycle" above) already returns `ok: true` independently of `canvas_ready`/`canvas_error`, but the rest of the stack didn't consistently honor that split:
  - `panel.js`'s `renderEmptyState()` never cleared `#msg` — a `canvas_error` banner from one attempt (e.g. "Timed out waiting for canvas to become ready") could freeze on screen through an unrelated later transition back to "nothing loaded" (a `/grc/close`, a failed auto-reopen, a fresh page load), live-reported as an old timeout banner sitting next to "No flowgraph loaded" and a disabled chat input. `renderEmptyState()` now calls `setMsg("", "")`.
  - `pollConversationState()`'s auto-reopen branch wrote `state.lastPath` only *after* its own `await fetch("/grc/open", ...)` resolved, and that call can legitimately take up to the canvas-ready deadline. Every 750ms tick firing before it resolved re-detected the same "path changed" condition and fired *another* `/grc/open` for the same path; since every `/grc/open` unconditionally kills whatever canvas is currently starting up (last-writer-wins, see "Process lifecycle" above), overlapping calls could repeatedly kill a freshly spawned canvas before it ever reached `/ready` — a self-inflicted livelock that manufactures its own timeout rather than merely reporting one. Fixed with an explicit `state.autoOpenInFlight` guard: set before the fetch, checked at the top of the function.
  - `_wait_for_canvas_ready` (`web.py`) swallowed every exception identically and never checked whether the canvas subprocess itself had already exited, so a crashed canvas (an uncaught exception in `get_gui_platform()`/`Application(...)`, which have no try/except around them, or `canvas_app.py`'s own `sys.exit(1)` when it can't find its `MainWindow`) still cost the full deadline before reporting the same generic message as a merely-slow canvas. It now also polls the passed-in `canvas_proc.poll()` each iteration and returns a `(ready, crashed)` pair, surfaced as "Canvas process exited unexpectedly — see canvas.log" instead of the generic timeout message when true.
  - `canvas_ready`/`canvas_error` used to reach the client only as a one-shot field on `/grc/open`'s own response. `refresh(forceCanvasReload, canvasReady)`'s `canvasReady` param *defaulted* to `true`, so any bare `refresh()` call made well after a real canvas failure (`doUndo`, `doRedo`, `doValidate`, the version-bump-triggered refresh in `pollConversationState`, or simply a fresh page load) would blindly try to re-point the iframe at a canvas already known to be dead — and, live-confirmed while writing this fix's own regression test, would show the distinct placeholder text but never the `#msg` banner at all, since that used to be set by each caller individually right after its own `refresh()` call returned, never by a bare call with nothing to say. Both fields are now mirrored into module-level `canvas_ready_state`/`canvas_error_state` (`web.py`, reset to `True`/`None` on `/grc/close`), re-exposed via `/grc/status`, and `refresh(forceCanvasReload, canvasReady, canvasError)` itself falls back to `statusRes.canvas_ready`/`statusRes.canvas_error` whenever the caller doesn't pass explicit values (an explicit pair — from `openGraph`/the auto-reopen path, which already have the freshest possible signal from their own `/grc/open` call — still takes priority) and calls `setMsg(...)` itself, positioned after its own `render()` call (the only thing that would otherwise clear a stale error-class `#msg`) so the message survives regardless of which caller triggered it.
  - The canvas placeholder's text was one static string ("No flowgraph loaded. Click Browse to choose one.") reused verbatim whether nothing was loaded or a flowgraph was loaded but the canvas failed — telling a user with a fully working chat session that nothing was loaded. Split into `CANVAS_PLACEHOLDER_EMPTY`/`CANVAS_PLACEHOLDER_CANVAS_FAILED`, applied via a dedicated `#canvas-placeholder-text` span rather than a raw text node.
  - `canvas_app.py`'s control-server bind retry budget (`start_control_server`) was only 3s (10×0.3s) — shorter than `web.py`'s own worst-case teardown of a prior canvas (`_terminate_canvas_proc`: up to 2s SIGTERM + 2s SIGKILL confirmation). If that budget was exhausted, `/ready` became permanently unreachable for the rest of that subprocess's life: the canvas could render perfectly fine and `_wait_for_canvas_ready` would still burn the full deadline and report a "timeout" that had nothing to do with the canvas itself. Bumped to 25 attempts (~7.5s).
  - The 20s canvas-ready deadline (`_wait_for_canvas_ready`) is now `GRC_CANVAS_READY_TIMEOUT`-overridable (mirroring `GRC_BROADWAY_PORT`/`GRC_CANVAS_CONTROL_PORT`/`GRC_AGENT_PORT`) specifically so tests can force a fast, deterministic `canvas_ready=false` instead of waiting out a real 20s — see `test_frontend.py`'s `canvas_failure_server`.

## Test Gate

```bash
uv run pytest tests/test_unit.py        # fast, no LLM
uv run pytest tests/test_web_app.py     # web endpoints, no LLM
uv run pytest tests/test_frontend.py    # panel.js, real Chromium via Playwright, no LLM
uv run pytest tests/test_integration.py # live model, ~15-20 min
uv run ruff check
```

`test_unit.py`/`test_web_app.py` still touch live network (lite.duckduckgo.com search, page fetches, Ollama embeddings/chat for RAG) — they are not fully hermetic, but need no GUI/display server. `test_integration.py` runs the full scenario suite against a live local model; expect occasional flakiness on weaker/quantized models (loop/retry limits, tool-call formatting) unrelated to code correctness — retry the specific failing scenario before treating it as a regression.

- **`test_frontend.py` exists because HTTP/string-level checks cannot catch `panel.js`'s state-lifecycle races** — an independent audit found this project had gone through six-plus rounds of "fix the frontend state machine" bugs with no automated test able to catch the next one, since every one of them (see the `preserveGrcOnReset`/`getChatFramePath` entries below) only ever reproduces with a real DOM, a real iframe, and real timers. It spawns a genuine `uvicorn` subprocess on its own port triple (19932/19085/19933 — distinct from the defaults, `conftest.py`'s ports, and `.claude/launch.json`'s isolated config, so it can run alongside any of them) and drives it with a real headless Chromium via Playwright. Requires `playwright install chromium` once after `uv sync` (the `playwright` pip package alone does not ship the browser binary). No live LLM backend needed — the vendor chat bundle's own conversation-id routing is simulated directly via `window.history.pushState` on the chat iframe (confirmed, by directly reading the shipped bundle, to be exactly the mechanism it uses itself), since `panel.js` never distinguishes a real vendor-issued conversation id from any other non-`"/"` pathname. When adding a test for a *new* timing-sensitive decision, prefer calling the relevant function directly inside one synchronous `page.evaluate()` (as `test_clear_history_overlay_hides_promptly_after_conversation_starts` does) over a wall-clock `wait_for_timeout` race against real `setInterval` timers — two independent timers' relative phase depends on exact page-load timing, not just their interval lengths, so a timeout-based race is flaky in a way a synchronous check isn't.
- **Resetting the chat widget must never unload the flowgraph.** `resetConversation()` (the "chat looks stuck?" button and the auto-heal for an unresolvable conversation id) calls `resetChatFrame(true)`, not `startNewConversation()` — the `true` sets `state.preserveGrcOnReset`, consumed once by `pollConversationState`'s transition-to-`"/"` handler to skip its `unloadGrc()` call. Live-confirmed this used to close the loaded file and kill its live canvas process on every auto-heal and every click of that button, with zero relation between "the chat widget looks broken" and "the loaded flowgraph should be abandoned," and no confirmation dialog (unlike Clear History). `startNewConversation()` (the "+" button, Clear History) is the one path that should still unload — it calls `resetChatFrame()` without the flag.
- **Any decision that must be correct within a tick or two of a real navigation must read the chat iframe's live `location.pathname` directly (`getChatFramePath()`), never `state.lastPath`.** `state.lastPath` is written only by `pollConversationState`'s 750ms tick; `integrateSettings` runs on its own independent 300ms tick and used to trust `state.lastPath` for the Clear-History-overlay and session-history-panel visibility decisions — live-measured at a real, fixed ~734ms window (not occasional) where Clear History showed over an already-active conversation after every single message sent, because the two timers' relative phase let the faster one read a value the slower one hadn't updated yet. `getChatFramePath()` bounds this to at most the single 300ms tick that owns the decision, not two independently-scheduled clocks compounding.
- **The dashboard's own URL is a real, restorable representation of the active conversation, not just `localStorage`.** `setUrlConvId()` mirrors the active conversation id into a `conv` query param via `history.replaceState` (not `pushState` — this tracks "what's currently shown," it isn't meant to grow browser history per conversation) every time `pollConversationState` detects a path change, and clears it inside `resetChatFrame()`. `restoreSession()` checks the URL's `conv` param before falling back to `localStorage`. This only ever touches the *parent* document's own URL/history entry — a same-origin iframe maintains its own independent `location`/history, so it cannot see or be affected by the parent calling `history.replaceState()` on itself.
- **A forced canvas failure is tested like any other lifecycle transition, not left to the 20s live deadline.** `test_frontend.py`'s `canvas_failure_server` fixture is a second `uvicorn` subprocess (its own port triple, 19942/19086/19943) launched with `GRC_CANVAS_READY_TIMEOUT=0.01`, so `_wait_for_canvas_ready` gives up before its very first `/ready` poll — deterministically reproducing `canvas_ready: false` without actually breaking `canvas_app.py` or waiting out a real timeout. `test_canvas_timeout_leaves_chat_enabled_and_clears_banner_on_close` opens a file against it, confirms chat stays enabled and the placeholder shows the distinct `CANVAS_PLACEHOLDER_CANVAS_FAILED` text (not "No flowgraph loaded"), then closes it and confirms the error banner actually clears. `test_poll_conversation_state_guards_against_duplicate_open` fires `pollConversationState()` a second time (via a direct `page.evaluate()` call, not a wall-clock wait on the real `setInterval` — same reasoning as this section's first entry) while an earlier call's `/grc/open` is still deliberately pending, and asserts only one such request reaches the server, confirming the `autoOpenInFlight` guard.
