# AGENTS.md

Rules for AI coding agents working on this codebase. Direct, data-driven, zero fluff.

## Architectural Vision & Core Rules

- **Simplify First**: Lean towards simplifying, not complicating. If a feature or approach is ad-hoc, hardcoded, or not essential, **remove it**.
- **No Brittle Reinventions**: Reject complex manual implementations or from-scratch logic when reliable, standard libraries can replace them. PydanticAI owns the agentic loop, tool dispatch, and message history; gbulb owns the asyncio+GTK event loop unification. Do not reimplement either.
- **No Backward Compatibility**: Delete dead code completely. Do not write shims, dual-format persistence layers, or legacy bridges. Keep changes clean and direct.
- **No Assumed Reasoning Failures**: Do not assume task failures are solely due to LLM reasoning. Audit the execution harness for context flooding, poor prompt construction, hidden ad-hoc logic, or silent error message clipping. Correctness lives at the source.
- **Maximizing Context & No String-Based Clipping**: Do not enforce arbitrary context limits beyond what the backend actually supports. Never clip inputs or outputs using raw character slicing which breaks structured context.
- **Be Bold, Objective, and Grounded**: Base every decision on grounded, verified observations, never on assumptions. Ask for clarification when requirements are ambiguous or when a major decision (e.g. library selection, backend config) needs to be made.
- **GUI-only. No CLI.** `grc-agent` is a single-entry-point native desktop app. `pyproject.toml`'s `[project.scripts]` must always point directly at `grc_agent.desktop_app:main`, launching the GUI immediately with no subcommands, no `argparse`, and no flags like `--check`/`--doctor`/`--version` beyond what GRC's own arg handling already does. If a startup-error or diagnostic need arises, solve it inside the GUI itself (e.g. a `Gtk.MessageDialog`, the sidebar's status bar) — never a command-line mode.

## Architecture at a glance

Single-process, single-thread native GTK3 desktop app. `gbulb.install(gtk=True)`
unifies asyncio and the GLib main loop — the agent, canvas, and all tool calls
run on one thread with zero cross-thread marshaling. GRC's own `MainWindow` is
extended in-place with a `ChatSidebar` widget (packed into the MainWindow's
existing `Gtk.HPaned` structure — no reparenting of the `DrawingArea`).

Everything lives in the installable `grc_agent` package, under `src/grc_agent/`.
It must never import from anywhere outside the package — the one accepted
exception is `ingest.py` reading the `docs/wiki_gnuradio_org/` corpus, which is
shipped package data via `pyproject.toml`'s `force-include`.

| File | Role |
|--------|------|

Wires toolbar signals (New, Blocks toggle), Ctrl+/- zoom, SIGTERM shutdown. Auto-detects the active graph from GRC's notebook — no Browse button. |

| `_paths.py` | Package-relative runtime-data directory resolution (`vectors_dir()`, `docs_dir()`). |
| `exec_monitor.py` | Detects flowgraph execution failures from GRC's native console message bus (`gnuradio.grc.core.Messages.register_messenger`) — parses `>>> Done (return code N)`/`Generate Error:` markers, ignores SIGTERM (`-15`, GRC's own Kill button) as a user-requested stop, buffers a bounded log (512KB cap via `deque`), retains the last completed run's output until the next run completes, and reports failures via callback to `ChatSidebar.notify_run_failure()` which sends a short agent notification (return code + tool hint). Wired at startup in `desktop_app.py`. |

Data flow: `.grc` file → GRC's `MainWindow` → `window.current_page.flow_graph` (shared between canvas and agent tools on the same thread) → `inspect_graph()` → JSON tool result.

## Engineering Rules

- **No hand-picked heuristics.** No per-field allowlists, per-scenario branches, regex routing, or prompt folklore. If logic is needed, it is one uniform rule applied to every case (see `keep_param` in `adapter/graph.py`).
- **Prefer native methods.** Use GNU Radio GRC's Python API — `param.hide`, `param.category`, `Block.is_variable`, `flow_graph.is_valid()`, etc.
- **Fix at the source.** Correctness lives in the tool/handler that produces data, not in a post-processor.
- **No silent transformation.** Any truncation, filtering, or omission in model-facing output must be explicit.
- **Simplify by removal.** Prefer deleting code over adding it.
- **Evidence before assertions.** Every claim cites a verified observation, never intent. A green test is necessary, not sufficient — inspect actual data flow.
- **Prefer pydantic_ai's own sanctioned extension points** (capabilities, `ModelRetry`, `AsyncTenacityTransport`, etc.) over hand-rolled retry/loop/context logic.
- **gbulb owns the event loop.** `gbulb.install(gtk=True)` must be called after `gi.require_version("Gtk", "3.0")` but before any asyncio/GTK usage. All async work (agent streaming, runner subprocess I/O) runs on this unified loop — no `GLib.idle_add` marshaling for widget updates from async code.
- **The agent and canvas share the same `FlowGraph` object.** In the single-process app, `NativeFlowgraphProxy` forwards to `window.current_page.flow_graph`. Agent edits modify it in-place — `after_agent_edit()` refreshes the canvas (`fg.update()`, `queue_draw()`, scrolls to new blocks), re-derives the poll's sync baselines (`last_synced_export_hash`, `last_disk_hash`, the state-cache version — see the safety-net poll convention below), and pushes to GRC's native `state_cache`; no reload-from-disk is needed. Manual edits need `write_flow_graph_atomic()` + `push_undo_snapshot()`.
- **Never bump the version number without explicit user approval.** The project version in `pyproject.toml`, `CITATION.cff`, and `CHANGELOG.md` must stay in sync and must NOT be changed by an AI agent unless the user explicitly asks. A version bump is a user decision, not an agent decision — even if the agent thinks the version looks stale or inconsistent.

## Tool Surface

Five model-facing tools. All are wired in `agent.py`'s `grc_tools()`,
registered as plain `Tool(fn, name=..., description=...)` relying on
pydantic-ai's own signature/type-hint introspection. The other two are
provider-adaptive `WebSearch`/`WebFetch` capabilities.

| Tool | Direction | Engine |
|------|-----------|--------|
| `inspect_graph` | read | `adapter.inspect_graph()` (Stage A + B filtered) |
| `query_knowledge` | read | `adapter.query_catalog()` (catalog domain) **or** `query_docs()` (docs domain) — builds its `.db` on first use via `ingest.py` if missing. Vector search (sqlite-vec) is primary; falls back to a local SQLite FTS5/BM25 keyword search when the embed call fails or no vector index exists, tagged via `search_mode`/`message` in the tool's JSON output. Model-controlled result count via `k` (default 5, clamped 1-20). |
| `web_search` | read | pydantic-ai `WebSearch` capability — native on OpenRouter, `adapter.lite_web_search` (async `ddgs` client) local fallback on Ollama. |
| `web_fetch` | read | pydantic-ai `WebFetch` capability — native where supported, bundled markdownify fallback otherwise. |
| `generate_python` | read | `adapter.preview_flowgraph_py()` — renders the flowgraph's generated Python source via GNU Radio's own `Generator`, in-memory only (`_build_python_code_from_template()`, never `.write()`); no disk I/O. Shares `generate_flowgraph_py`'s validity/hier-block/C++ gate (`_check_codegen_preconditions`) but does **not** apply its `run_options` override, so it reflects the flowgraph's actual configured output. Model-controlled cap on block-source files via `k` (default 5, clamped 1-20, same convention as `query_knowledge`) — the main flowgraph script is always kept regardless of `k`; any dropped Embedded Python Block/Module files are counted in `omitted_files`, never silently. A failure (invalid graph, hierarchical block, C++ output, or an unsupported GNU Radio internal API) raises `ModelRetry`. |
| `change_graph` | write | `adapter.change_graph()` — 7-phase transactional batch mutation with rollback; a failure raises `ModelRetry` instead of returning `ok=false` |
| `get_run_log` | read | `exec_monitor.get_last_run_log()` via `NativeFlowgraphProxy.get_run_log()` — returns the last completed run's stdout+stderr (success or failure). Retained until the next run. When a run fails (non-zero, non-SIGTERM return code), the agent is automatically notified with the return code and prompted to call `get_run_log` for the full output — replacing the old Yes/No fix-error bubble that injected the full log as a prompt blob. |

## Key Conventions

- **Param filtering** (one rule, in `adapter/graph.py`'s `keep_param`): Stage A (every mode) drops `dtype == "id"`, `showports`, `bus_structure_*`, `hide == "all"`, `dtype == "gui_hint"`. Stage B (overview mode only) keeps `hide == "none"` OR `dtype == "enum"` with a non-default value or structural status OR `value != default` OR references a variable OR the param is type-controlling OR `generate_options`.
- **State values:** `enabled`, `disabled`, `bypass` (accept `bypassed` as alias). Validated against the block's own `STATE_LABELS`.
- **Block lookup:** use native `flow_graph.get_block(name)`, not a manual scan.
- **Atomic save:** temp file → fsync → `os.replace()` → directory fsync. Lock via `fcntl.flock` on `.grc_agent/<name>.lock`.
- **`change_graph` output:** returns `{"ok": true}` on success; `{"ok": false, "error_type": "...", "errors": [...]}` on failure. The tool wraps this: on `ok=false` it raises `ModelRetry` with errors folded into the message. `force=True` bypasses native-validation failures only.
- **Graph-validity gate:** `agent.output_validator(validate_flowgraph_state)` runs on every turn's final output. If any `change_graph` tool call appears in that turn's message history, it checks the live flow graph's native `is_valid()`/`iter_error_messages()` and raises `ModelRetry` if invalid.
- **`flow_graph.validate()` must be called before trusting `is_valid()`/`iter_error_messages()`.** GNU Radio's `Element._error_messages` list is populated only by an explicit `.validate()` call and is cleared — without being refilled — by `.rewrite()`. Since every mutation path calls `rewrite()`, skipping the `validate()` call right before checking `is_valid()` makes the check vacuously pass.
- **Ports** (`inspect_graph`'s block `inputs`/`outputs`): Stage A drops hidden ports. Stage B additionally drops a port only if it is both `optional` and unconnected.
- **`type`/dtype auto-resolution:** `change_graph` accepts the literal string `"auto"` on a type-controlling param, resolved from a connected neighbor's dtype via `resolve_auto`. If a new block's type is `"auto"` but it has no already-typed neighbor to resolve from — either no connection in the batch, or only other `"auto"` blocks — the batch fails loudly with an `auto_resolve_failed` error (and rolls back) instead of silently keeping GNU Radio's own default.
- **`set_param`'s "unknown param" error lists the block's actual valid param keys** — so the model doesn't waste a `query_knowledge` round-trip guessing.
- **`resolve_auto` only ever resolves from an explicit, non-`"auto"` value** — it never guesses from an equally-unresolved neighbor, and it fails loudly instead of silently defaulting.
- **`change_graph`'s tool description is derived from its own docstring** (`docstring_format="google", require_parameter_descriptions=True`).
- **`change_graph` calls `flow_graph.rewrite()` twice** in its success path — once conditionally right after Phase 5 (`resolve_auto`), only if `add_blocks` is non-empty, since GNU Radio's `Block.rewrite()` is what populates a newly-added block's `active_sinks`/`active_sources`, which Phase 7's connection lookup (`_find_port`) needs; and once unconditionally at the end, after Phase 7 (`add_connections`), to settle all derived state before the validation gate. Both are load-bearing — the old *per-block* `rewrite()` inside the `add_blocks` loop (one call per new block) was removed and replaced by these two batch-level calls.
- **The conditional Phase-5 rewrite being gated on `add_blocks` (not on `update_params`) has a real consequence for `epy_block`/`epy_module`.** `EPyBlock.rewrite()` (GNU Radio's own code) re-parses `_source_code` and regenerates that block's ports/params only when *that block's* `rewrite()` runs — which, for a call with no `add_blocks`, only happens in the final unconditional rewrite, i.e. *after* Phase 7's `add_connections` already ran. Confirmed live: changing an existing epy_block's port count in the same call as connecting to the newly-implied port fails outright (`_find_port` can't find a port that doesn't exist yet); changing a port's *dtype* (same port key, e.g. complex→float) in the same call as connecting to it is worse — `_find_port`/`connect()` succeed against the *old* port object, then the final rewrite's `_update_ports` builds a new port object and disconnects the old one, silently dropping the connection Phase 7 just made. A dedicated post-rewrite check now catches this — and, per an adversarial-testing pass that found the first version of this check was itself incomplete on two fronts, does so correctly:
  - **Scope**: it snapshots `set(flow_graph.connections)` once, before *any* phase-3-onward mutation or rewrite (not just the connections `add_connections` itself made), and compares that snapshot against the post-final-rewrite state. An earlier version only tracked connections made in the *same call's* `add_connections`, so a call with *only* `update_params` (no `add_connections` at all) silently missed an already-existing connection dropped the same way — confirmed live, this used to return `ok: true` while a pre-existing connection vanished.
  - **Identity, not strings**: it compares actual `Connection` objects (via set membership — `Connection.__eq__`/`__hash__` are keyed on `(source_port, sink_port)`, and `Port` has no override, so this is genuinely object-identity-based), not `(block_name, port_key)` string tuples. GNU Radio can rekey a port **in place** (`Port.rewrite()` sets `self.key = self.name` on the *same* object whenever a port's dtype becomes `"message"` — e.g. a `pad_sink` reconfigured to `type='message'`), which a string-tuple comparison would misread as a dropped connection and roll back a perfectly valid batch. Comparing objects is immune to an in-place attribute mutation.
  - Rolls back the whole batch via the same unconditional `errors` list `remove_connections`/`add_blocks` failures already use — deliberately *not* just under `force`, since a connection vanishing without a word is never acceptable regardless of `force` (which only bypasses GNU Radio's own general validity opinion, not this).
  - The correct workaround, reflected in the system prompt: change an existing block's code/ports in its own `change_graph` call (no `add_blocks` in that same call), confirm the new shape via `inspect_graph`, then wire/param-set in a follow-up call — or add the block and its final code in the *same* call, since any non-empty `add_blocks` unlocks the earlier Phase-5 rewrite regardless of what's being added.
  - `change_graph_func`'s `ModelRetry` message (`agent.py`) only suggests `force=True` when `error_type == "validation_failed"` — every other failure code in the general `errors` list (including `connection_silently_dropped`) is rolled back unconditionally, so suggesting `force` there would waste one of the model's limited retries on something that can never succeed.
- **Undo/redo uses GRC's native StateCache.** Our custom disk-based undo/redo stack (`adapter/snapshots.py`) is kept for snapshot pushes during `sync_manual_edit` and `change_graph`, but the UI buttons and `NativeCanvasManager.undo()`/`redo()` methods are removed. GRC's built-in Ctrl+Z/Y works natively.
- **New blocks are positioned by a headless, collision-avoiding placement.** `change_graph`'s `add_blocks` phase uses `grandalf`'s Sugiyama-style rank assignment + a spiral grid search for the non-overlapping coordinate. This must stay fully automatic since `inspect_graph` filters coordinates out of context.
- **Vector DBs are built, not shipped.** `adapter._ensure_db_built` lazily calls `ingest.py` to build the catalog/docs DB on first `query_catalog`/`query_docs` call. Each backend gets its own `.db` file. `_db_meta` table stores `embedding_model` and `corpus_version`; a mismatch (or a missing lexical FTS5 index) triggers a rebuild. A per-process `_FRESHNESS_CACHE` (`rag.py`) short-circuits this metadata check entirely once a `(domain, db_path, model)` combo has been verified fresh, so it's no longer re-checked on every single query within a process's lifetime — only on the first query and after an actual rebuild. The OpenAI embed client is cached at module level for connection reuse. If every embed call fails (embedding backend unreachable), the DB is built lexical-only (FTS5, no vector index) rather than failing the whole ingest — see `query_knowledge`'s Tool Surface row.
- **Chat model/provider is user-configurable AND live-swappable.** `agent_factory.build_interactive_agent()` reads `settings.py`'s `load_settings()` to build the model. Three providers: `ollama` (local), `ollama_cloud` (cloud), `openrouter` (cloud). When the user changes provider/model/key in the Settings dialog and hits Save, `chat_sidebar.py`'s `_apply_settings_save` writes to `.env`, runs `preflight_connection` against the new backend, then invokes `sidebar._rebuild_agent` (wired in `desktop_app.py:build_app` to `lambda: build_agent_from_cfg(load_settings())`) to rebuild the Agent **in-place** and call `sidebar.set_agent(new_agent)` — the running chat process picks up the new provider immediately, no restart needed. The `_message_history` is preserved verbatim across a swap (ModelMessage objects are provider-agnostic). A preflight failure surfaces a Yes/No "Save anyway?" confirm. The active provider is shown live in the toolbar's `provider-badge` (driven by `set_active_provider`, refreshed from `set_agent`). If `_rebuild_agent` is None (headless/test), the old restart-gated status message is used as a fallback. `extra_body={"think": True}` is recomputed on each rebuild — only sent to Ollama-based providers.
- **`generate_flowgraph_py` (Run/Stop codegen) overrides `run_options` to `'run'` — MUST call `.rewrite()` after `.set_value()`.** `FlowGraph.get_option(key)` returns `params[key].get_evaluated()`, which reads a *cached* `Param._evaluated` — and `.set_value()` only sets `self.value`, it never touches that cache. Only `.rewrite()` resets and recomputes `_evaluated`. Without the `.rewrite()` call, the generated `no_gui` script silently still contains `input('Press Enter to quit: ')`.
- **`preview_flowgraph_py` (the `generate_python` tool) is genuinely read-only** — it calls GNU Radio's `TopBlockGenerator._build_python_code_from_template()` directly, which is pure computation with no file I/O; `write()` is the separate wrapper that actually opens/writes files, and is never called from this path (verified by direct testing: no new files appear after a preview call). It deliberately skips `generate_flowgraph_py`'s `run_options` override, so it reflects the flowgraph's real configured output rather than the Run/Stop-specific one. Returns one entry per generated file — the main script plus one per Embedded Python Block/Module instance, if present, since GNU Radio's generator writes those as separate files; GNU Radio always appends the main script last, so it's kept unconditionally and only the block-source entries are capped by `k` (`omitted_files` reports the drop count). Each entry's `path` is informational only — where GRC would write it, not a real file — since at least one model has been observed formatting it as if it were a downloadable link. Hierarchical-block flowgraphs (`generate_options` starting with `hb`) are rejected outright, not just for `generate_flowgraph_py`: `HierBlockGenerator.__init__` does a real `os.mkdir()` as a side effect of construction, so there is no side-effect-free path to preview one. Reaching into `_build_python_code_from_template` (a private GNU Radio method) is a real fragility risk if a future GNU Radio version renames/removes it — an `AttributeError` there is caught and turned into a clear `ValueError` (surfaced as `ModelRetry`) rather than a confusing raw traceback.
- **GRC's Notebook/ScrolledWindow hardcodes a 600×400 minimum size** (`Notebook.py:123-124`). `NativeCanvasManager.setup_signal_handlers()` calls `set_size_request(1, 1)` on the `ScrolledWindow` to let the canvas shrink to whatever the paned divider allows.
- **Middle-click (button 2) canvas panning** binds on the `DrawingArea` and returns `True` to consume. GRC's own `DrawingArea.py` only handles button 1 (select/drag) and button 3 (context menu).
- **A 1.5s safety-net poll catches edits that don't fire a trackable GTK signal** (properties dialog OK/Apply/Cancel, context-menu actions). `native_canvas.py`'s `_check_for_unsynced_edit` compares `flow_graph_content_hash` against `last_synced_export_hash` and triggers `sync_manual_edit()` on mismatch. Most ticks skip that expensive comparison entirely via a cheap gate on GRC's own `page.state_cache` version tuple (`_state_cache_version`) — the full hash only runs when that tuple has moved, or every `_POLL_FULL_CHECK_EVERY` ticks (10, ~15s) regardless, as a backstop against edit paths that don't touch `state_cache` (e.g. block-library drag-and-drop add) or the rare undo-then-different-edit tuple collision.
- **The active flowgraph is resolved dynamically** (`NativeFlowgraphProxy` in `native_canvas.py`). `drawing_area`, `path`, and `current_flow_graph` are properties that resolve from `window.current_page` on every access — tab switches are followed automatically. `get_state_lock()` returns `None` (single-thread, no races). `notify_edit()` calls `canvas_manager.after_agent_edit()` directly.
- **`lite_web_search` is async** — uses `ddgs` (DuckDuckGo's official Python client) via `asyncio.to_thread` to avoid blocking the gbulb event loop. The old `lite.duckduckgo.com` HTML scrape was blocked by a CAPTCHA challenge as of 2026-07.
- **Settings use `dotenv.set_key`/`dotenv_values`** — the old manual regex-based `.env` upsert was replaced with `python-dotenv`'s built-in API.
- **`_sha256_file` uses `hashlib.file_digest`** — streams the file instead of loading it entirely into memory.
- **`_scroll_to_new_blocks` multiplies coordinates by `zoom_factor`** — scroll targets are in zoomed space.
- **`get_run_log` reads the last run's output from `exec_monitor`.** The monitor retains `_last_run_log`/`_last_run_code` (success or failure) until the next run starts. When a run fails (non-zero, non-SIGTERM return code), `ChatSidebar.notify_run_failure` sends a short agent notification with the return code — the full log is NOT injected as a prompt; the agent calls `get_run_log` as a structured tool result. The tool is wired through `NativeFlowgraphProxy.get_run_log()` (which bypasses `__getattr__` forwarding via `object.__getattribute__`) — no module-level singleton.
- **Incremental `force=True` editing is a taught strategy.** The system prompt explicitly tells the agent it may split a large edit across multiple `change_graph` calls in one turn, using `force=True` on intermediate calls that leave a required port temporarily unconnected — the environment re-validates only the final state at turn end (`validate_flowgraph_state` in `agent.py`). Structural error messages (`add_connection_failed`, `auto_resolve_failed`) include port dtype details so the model can diagnose mismatches and decide whether to split, insert a converter, or change a type.

## Test Gate

```bash
uv run pytest tests/test_unit.py              # fast, no LLM; needs a display (xvfb-run)
uv run pytest tests/test_exec_monitor.py      # fast, no LLM, no display
uv run pytest tests/test_isolation.py         # settings/model isolation, no LLM (one test makes a live Ollama Cloud call if OLLAMA_CLOUD_API_KEY is set)
uv run pytest tests/test_button_integration.py # tool/button integration, Ollama Cloud
uv run pytest tests/test_integration.py       # live model scenarios, Ollama Cloud, ~15-20 min
uv run ruff check
```

`test_unit.py`/`test_isolation.py` touch live network (lite.duckduckgo.com search,
Ollama embeddings/chat for RAG) — they are not fully hermetic. `test_isolation.py`
needs no GUI/display server; `test_unit.py` builds real GTK widget trees and needs
one (`xvfb-run`). `test_integration.py` defaults to Ollama Cloud (the project's
standard real-LLM test backend) when `OLLAMA_CLOUD_API_KEY` is configured — it
does not also run against local Ollama/OpenRouter just because they happen to be
available too. Force a specific backend with `GRC_TEST_BACKEND=ollama|ollama_cloud|openrouter`.

Tests that need a real GTK widget tree require `xvfb-run` (e.g. `xvfb-run -a uv run pytest`).
